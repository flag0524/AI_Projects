## Problem Statement

마네킹/옷걸이 제품 사진을 자연스러운 모델 착용 컷으로 변환하는 시스템이 이미
가동 중이다. 생성형 경로는 두 백엔드 — Leffa(HF Spaces), IDM-VTON(Replicate) —
와 절차적 폴백(`laongen_engine.generate_model_shot`)을 `GEN_BACKEND` 환경변수로
선택하는 프로바이더 추상화 위에 있다.

이 작업은 **세 번째 생성형 프로바이더로 Higgsfield를 추가**한다. 동기는 교체가
아니라 **기능 확장**이다: Higgsfield는 정지 착용 컷 생성(Marketing Studio
"UGC Virtual Try On")뿐 아니라, 이후 image-to-video(워킹·회전·마케팅 클립)로
확장할 진입점을 제공한다. 이번 범위(웨지)는 정지 착용 컷까지다.

## Demand Evidence

- 내부 동기: "기능 확장(영상 등)" — 기존 정지 try-on 위에 Higgsfield가 잘하는
  생성 역량(영상·포즈·마케팅 컷)을 얹는 것이 목표.
- 그린라이트 데모 기준: "의류 1장 → 모델 착용 컷"을 Higgsfield가 직접
  만들어내는 것. 며칠 안에 만들 수 있고, "Higgsfield가 직접 만든다"는 데모
  서사가 깔끔하다.
- 주의: 이것은 외부 고객의 검증된 수요가 아니라 내부 스폰서 승인을 겨냥한
  가설이다. 데모의 성공 = fidelity(아래 Open Questions) 확인.

## Status Quo

현재 `POST /api/v1/generate`는 `generate_model_shot`을 호출하고, 이는
`GEN_BACKEND`(hf/auto → HF Leffa, 그 외 → Replicate IDM-VTON, 둘 다 불가 시
절차적 폴백) 순으로 시도한다. `/generate/status`가 활성 백엔드를 보고한다.
프로바이더 계약은 사실상 두 함수다: `is_available() -> bool`,
`generate_tryon(...) -> PIL.Image`.

영상은 현재 시스템에 없다. 마케팅 클립은 별도 수작업.

## Target User & Narrowest Wedge

- 사용자: 내부 스폰서/VP(그린라이트 결정자) + 이를 운영할 콘텐츠/커머스 팀.
- 웨지: **의류 1장 + 모델 템플릿 → Higgsfield가 생성한 착용 컷 1장**.
  영상은 명시적으로 다음 단계로 분리(이번 범위 밖).

## Constraints

- 프로덕션 백엔드(FastAPI)는 **Higgsfield REST API/SDK를 직접 호출**한다.
  MCP는 런타임 경로가 아니다 — MCP는 개발·프로토타이핑·품질검증 단계에서
  에이전트(Claude)가 Higgsfield를 조작·모델탐색·프리셋 검증하는 통로다.
- 기존 프로바이더 추상화(`is_available` / `generate_tryon`)와 폴백 체인을
  그대로 유지하고 깨지 않는다 (CLAUDE.md §3 Surgical Changes).
- 인증: 새 환경변수(예: `HIGGSFIELD_API_KEY`). 키 없으면
  `is_available() == False` → 기존 백엔드로 자동 폴백.
- Windows 사내망 SSL 가로채기 환경 — 기존 `truststore` / CA 번들 패턴
  (`generative_provider._resolve_verify`)을 재사용한다.

## Premises

1. Higgsfield는 Leffa/IDM-VTON을 교체하는 게 아니라 확장 레이어다. — 동의
2. [핵심] "MCP 기반" = 프로덕션은 REST API 직접 호출, MCP는 개발/프로토/품질
   검증 통로. — 동의 (확정됨)
3. 생성형 모델은 캐릭터·스타일 일관성은 강하나 의류의 정확한 로고·패턴·핏
   보존은 전용 VTON보다 약할 수 있다 → 웨지의 1순위 검증 항목. — 동의
4. 웨지 = 정지 착용 컷 먼저, 영상은 다음 단계. — 동의

## Approaches Considered

### Approach A: Higgsfield 프로바이더 추가 (선택됨)
기존 파이프라인에 `higgsfield_provider.py`를 추가. `hf_provider` /
`generative_provider`와 동일한 `is_available()` / `generate_tryon()` 계약.
프로덕션은 Higgsfield REST API(Marketing Studio "UGC Virtual Try On")로 의류→
착용 컷. `GEN_BACKEND=higgsfield`로 선택. 개발 중에는 MCP로 모델·프리셋·품질
검증.
- Effort: S/M · Risk: Med (fidelity 미검증)
- Reuses: provider 추상화, laongen_engine 폴백 체인, SSL 패턴, /generate 라우팅

### Approach B: VTON(fidelity) + Higgsfield(모션/마케팅) 2단계
정지 컷은 기존 VTON 유지, 결과를 Higgsfield image-to-video로 후처리.
- Effort: M/L · Risk: Low-Med
- 보류 이유: 웨지가 정지 컷이고 데모 서사가 "Higgsfield가 직접 만든다"여서
  이번 범위와 어긋남. 단, 영상 단계에서 B가 가장 유력 후보로 복귀할 것.

### Approach C: Higgsfield 네이티브 (Reference Elements)
의류를 Element로 등록 → Nano Banana Pro/Seedream에 `<<<element>>>`로 착장 생성.
- Effort: M · Risk: Med-High (fidelity 가장 불확실)
- 보류 이유: 가장 미검증. 단, A의 Marketing Studio 경로가 fidelity 미달이면
  C의 Element 경로가 fallback 실험 대상.

## Recommended Approach

**Approach A.** 기존 추상화에 정확히 들어맞아 최소 diff로 며칠 내 데모가 되고,
"Higgsfield가 착용 컷을 직접 만든다"는 스폰서 데모 서사가 깔끔하다. 영상은
승인 후 별도 단계(B/C 재평가)로 분리한다.

### 구현 스케치 (코드 아님, 합의용)

1. `backend/pipeline/higgsfield_provider.py` 신설

   - `generate_tryon(garment_img, model_template, garment_desc, category, ...)`
     — `generative_provider.generate_tryon` 시그니처에 맞춤

     Try On"(또는 모델 탐색으로 확정된 엔드포인트) 호출 → 결과 이미지 다운로드
     → PIL 반환
   - SSL은 `_resolve_verify()` 동일 패턴 재사용
   - 실패 시 `GenerativeUnavailable` 계열 예외로 폴백 유도
   - 카테고리 매핑(top/bottom/dress/accessory → Higgsfield 입력) 추가
2. `laongen_engine.generate_model_shot` — `GEN_BACKEND in ("higgsfield",
   "auto")` 분기 추가, 기존 폴백 순서 보존
3. `backend/api/generate.py` `/generate/status` — `higgsfield_available`
   플래그와 활성 백엔드 문자열 추가
4. 변경은 위 3개 지점으로 한정 (Surgical)

# Engineering Review (2026-06-08, /plan-eng-review)

## 확정된 결정 (Locked Decisions)

1. **비동기 계약** — Higgsfield 생성은 길 수 있어 동기 HTTP는 프록시·게이트웨이
   타임아웃으로 깨진다. `POST /api/v1/generate`는 job_id 반환,
   `GET /api/v1/generate/result/{id}`로 진행중/완료/실패 폴링.
   (기존 HF/Replicate 동기 경로는 보존 — 둘을 같은 엔드포인트로 통합하는 설계
   필요. 아래 Failure modes 참조.)
2. **단일 의류만 (2026-06-08 개정)** — Leffa baseline에서 순차 2벌 체이닝이
   drift로 실패(블라우스·데님 모두 소실). 따라서 웨지는 **의류 1장 → 착용 컷**
   으로 확정. 순차 체이닝은 NOT in scope로 이동. (Higgsfield 복구 후 2벌 케이스가
   Leffa보다 나으면 재평가, 또는 단일-호출 다중입력/VTON 경로로 재설계.)
3. **폴백 캐스케이드** — higgsfield 실패 시 HF Leffa → Replicate → 절차적
   순으로 캐스케이드. 이를 위해 `laongen_engine.generate_model_shot`의
   `if backend in (...)` **배타 게이트를 "우선순위 순서대로 시도" 체인으로
   리팩터**한다 (make-the-change-easy: 리팩터 먼저, 기능 추가 나중).
4. **전체 테스트 스위트** — 저장소에 테스트 인프라가 없음. pytest 도입 +
   REST 모킹으로 15개 코드패스 전부 커버.
5. **폴링 전략** — 고정 간격(2~3초) poll + 최대 폴링 횟수/총 타임아웃 상한.

|---|---|

| `GEN_BACKEND` 선택 + 폴백 (`laongen_engine`) | 캐스케이드 체인으로 리팩터(결정 3) |

## NOT in scope (명시적 연기)

- **영상(image-to-video)** — 정지 컷 웨지 검증 후 별도 단계(Approach B 재평가).
- **순차 다중 의류 체이닝(상의→하의)** — Leffa baseline에서 drift 실패 확인
  (2026-06-08). 단일 의류만 지원. Higgsfield 단일-호출 다중입력 또는 VTON-합성
  경로로 재설계 시 재검토.
- **비용 가드레일(`get_cost` 프리플라이트) / 지수 백오프** — 운영 비용이
  문제로 드러나면 그때. 지금은 고정 간격 + 타임아웃 상한으로 충분.
- **Higgsfield 네이티브 Reference Elements 경로(Approach C)** — A가 fidelity
  미달일 때만 실험.
- **job 상태 영속화(SQLite/Redis)** — 이번은 in-memory dict + uvicorn 단일
  워커로 제약. 멀티워커/재시작 내구성은 운영 단계에서. **배포 제약: 단일
  워커 필수**(`--workers 1`), 아니면 result 조회가 조용히 실패.
